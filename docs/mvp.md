# ClearLine MVP 路线

ClearLine 第一阶段坚持使用现有虚拟音频线 / 虚拟麦克风设备，不把自研 Windows 虚拟音频驱动作为 MVP 目标。

## 阶段 1：枚举麦克风 + 直通音频管线

目标：建立可运行的本地音频管线基础。

- 枚举 Windows 输入设备。
- 允许用户选择真实麦克风。
- 建立输入 stream、处理回调、输出 stream 的基础结构。
- 使用 `BypassSuppressor` 做无降噪直通测试。
- 显示输入电平。
- 保持 UI 简单：设备选择、降噪强度、启动/停止、状态提示。

当前仓库已完成阶段 1 的项目骨架、接口抽象、Windows 输入设备枚举、输入 stream 启动、输出 stream 启动、环形缓冲直通、启动预缓冲、基础声道转换、实时路径缓冲复用、固定帧切片器、可选抗风噪前处理、实时输入电平显示、处理链路诊断显示和缓冲状态显示。

## 阶段 2：低延迟路线归档

目标：保留低延迟方案的技术积累，但不再作为当前用户路径。

- `LowLatencySuppressor` 和 `rnnoise` feature 保留在 `clearline-core` 中，作为 legacy/dev 对比测试入口。
- 默认 `clearline-app` 不启用 RNNoise，不再显示“低延迟降噪”入口，也不再把 RNNoise 作为高质量模式回退。
- 旧版设置中的 `low_latency` 会自动迁移到 `high_quality`。
- `BypassSuppressor` 继续仅作为内部回归测试和问题定位路径。

当前实测结论：DeepFilterNet + 回音消除效果更稳定，RNNoise 在部分设备/噪声场景下会引入杂音。因此用户可见路线收敛到 DeepFilterNet，低延迟路线暂不继续作为产品主线。`抗风噪增强` 前处理仍保留，可按需要在 DeepFilterNet 前启用。

## 阶段 3：接入高质量降噪

目标：提供更高质量但可能更高延迟的模式。

- 优先评估 DeepFilterNet 或类似高质量方案。
- 将高质量实现放在 `HighQualitySuppressor` 后面。
- 明确模型加载、推理线程、缓冲策略和延迟预算。
- UI 中只暴露降噪强度，不再显示降噪模式选择，也不暴露复杂模型参数。

当前用户路径以 `DeepFilterNetModelBundle` + `deepfilternet-tract-worker` 为准。打包模型目录固定为 exe 旁边的 `models/deepfilternet`，要求包含 `enc.onnx`、`erb_dec.onnx`、`df_dec.onnx` 和 `config.ini`。设备页不再暴露手动模型目录；状态页显示打包模型是否可用、当前后端、推理延迟、队列水位、迟到帧和丢弃帧。

当打包模型有效时，启动后会选择 `deepfilternet-tract-worker` 后端。模型缺失、不完整或加载失败时，正式管线会拒绝启动并提示 DeepFilterNet 模型不可用，不再回退到 RNNoise。UI 提供统一的 `柔和 / 标准 / 强力` 降噪强度；当前默认路线是 DeepFilterNet + 回音消除。下一步需要根据真实设备测试继续调队列容量、迟到帧补偿策略和模型资源发布方式。

虚拟麦克风主路径已支持非 48 kHz 录音设备：麦克风输入和系统播放参考音频会先通过 `rubato 0.14` 重采样到 48 kHz 处理域，再进入回音消除、抗风噪、DeepFilterNet 和 48 kHz 虚拟麦克风输出。44.1 kHz 真实设备当前仍缺少实机验证；已有单元测试覆盖 44.1 kHz / 96 kHz / 48 kHz 转换行为。

## 阶段 4：输出到虚拟音频线

目标：把处理后的音频输出到用户已有的虚拟设备。

- 枚举可用输出设备。
- 允许用户选择类似 VB-CABLE、Virtual Audio Cable 等 Windows 虚拟线的输出端。
- 把处理后的音频写入该输出设备。
- 不在当前阶段编写 VB-CABLE 使用引导；VB-CABLE 仅作为开发测试用输出设备。
- 不在此阶段开发内核态虚拟音频驱动。

当前仓库已完成输出设备枚举、UI 选择入口和基础直通输出。普通音频设备直通输出会让输出 stream 使用输入采样率，并把输入声道转换为输出设备默认声道数，避免输入/输出默认格式不一致造成明显电子音。虚拟麦克风 / VB-CABLE 主路径会把非 48 kHz 输入重采样到 48 kHz，匹配当前虚拟麦克风输出契约。输出端启动时会等待约 20ms 预缓冲，避免启动瞬间误报欠载。状态页会显示缓冲水位、估算缓冲延迟、算法帧延迟、欠载样本和溢出丢弃样本，并给出中文诊断建议。后续仍需要补充所有输出后端的更完整格式协商和输出延迟策略。VB-CABLE 仅作为当前开发测试用输出设备，不把 VB-CABLE 使用引导作为产品文档目标。

## 阶段 5：Windows 默认录音设备设置

目标：减少用户手动配置成本。

- 封装 Windows 默认录音设备设置接口。
- 优先把 Windows API 细节隔离到单独模块或 crate。
- 如果 API 权限、系统版本或设备类型存在不确定性，先通过接口返回可解释错误，不在 UI 层硬编码假设。
- 保留手动设置说明作为回退路径。

## 非目标

- 账号系统。
- 云端处理。
- 多麦克风混音。
- 直播声卡功能。
- 第一阶段自研 Windows 虚拟音频驱动。
